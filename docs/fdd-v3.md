> This file records behavioral drift between the V1/V2 FDDs and the 1.0 implementation.
> Its purpose is to serve as a correction layer when merging fdd-v1.md and fdd-v2.md into a single authoritative spec.
> Each section names what the prior FDD said, what actually shipped, and how the merged spec should read.

# UI Layout

## Console placement
**V1 said:** Console is a tab in the right column, alongside Log and NPCs.

**V2 said:** Console is SSE-only, fundamentally transient, no GET needed.

**Actual:** Console is a persistent resizable bottom drawer in the left column, always visible beneath the Input/Additional Context tabs. It is not a tab at all. The right column contains no console panel.

**Merged spec:** Console is a persistent footer panel at the bottom of the left column. It is always visible and resizable. It receives SSE events via the runtime `console` stream. No GET endpoint is needed; transient output is not persisted beyond the process lifetime.

---

## Log tab
**V1 said:** Right column has a "Log" tab showing persisted standard-assistant transcript output, with structured Q&A sections, a table of contents, and support for browsing saved sessions.

**V2 said:** (no explicit mention — V2 introduced sessions but did not formally retire the Log tab concept)

**Actual:** No Log tab exists. Transcript browsing happens in the right-column mode tabs via the session selector. The Assistant tab renders the full session feed for the selected session, including a sticky table of contents when there are two or more exchanges.

**Merged spec:** The Log tab no longer exists as a distinct UI element. Session browsing and transcript viewing are served by the session selector dropdown and the mode-specific feed view (Assistant feed or NPC exchange feed).

---

## Mode selector location
**V1 said:** A mode selector (`Standard` / `NPC Generator`) lives on the Input tab in the left column.

**V2 said:** (implicitly: sessions are pre-filtered by kind and only visible in each mode tab — the mode tab *is* the selector)

**Actual:** Mode selection is the right-column tab system, with tabs labelled "Assistant" and "NPC Cards". There is no mode selector control on the Input tab. The Input tab's prompt area and controls apply to whichever right-column tab is active.

**Merged spec:** Mode selection is the right-column tab system. The left-column Input tab always reflects the active mode.

---

## Right-column tab structure
**V1 said:** Right column has three tabs: Console, Log, NPCs.

**V2 said:** Sessions are visible in the mode tabs; the mode-tab structure replaces V1's right-column layout (implied, not spelled out).

**Actual:** Right column has two mode-based tabs: "Assistant" and "NPC Cards". Each tab contains a session selector at the top. The Console and Log tabs from V1 do not exist. The NPCs tab from V1 is replaced by the NPC Cards tab, which holds both the cards grid and the exchange feed.

**Merged spec:** Right column has two mode tabs: "Assistant" and "NPC Cards". Each tab includes a session selector and New Session button, followed by the mode-specific content area.

---

# Transcript Persistence

**V1 said:** Standard-assistant transcripts are stored as files under `logs/`. Session files are created lazily after the first successful assistant answer. Filenames begin with a timestamp and a model-provided session title. Contents preserve ordered Q&A exchanges, including retrieval progress entries. Older logs remain browsable from the Log tab.

**V2 said:** (inherits V1, extended with sessions concept)

**Actual:** No file-based transcript storage exists. All transcripts are stored in the SQLite app database (`sessions` and `sessionEntries` tables). No `logs/` directory is created. Session titles are still generated by the model on the first completed response. Older sessions are browsed via the session selector dropdown, not a file-backed Log tab. The `Runtime File Locations` section in V1 listing `logs/` is no longer accurate.

**Merged spec:** Transcripts are stored in the app database. The `logs/` directory and file-based transcript format are retired. Session titles are model-generated and durable once the first response completes. Session browsing uses the in-app session selector.

---

# NPC Generation

## NPC identifier control
**V1 said:** Each NPC card requires a numeric identifier. If the model returns an existing numeric identifier, that card is updated in place. New cards must use identifiers above the current maximum saved identifier. The model controls which IDs to assign or reuse.

**V2 said:** (inherits V1 for this behavior)

**Actual:** NPC IDs are auto-assigned by the database (SQLite auto-increment). The model never supplies or controls IDs. Every NPC run creates new records; there is no update-in-place path. The model-controlled ID assignment and revision rules from V1 are completely absent.

**Merged spec:** NPC IDs are assigned by the database. The model does not specify IDs. Each run always creates new NPC records. In-place revision is not supported; a new run produces new cards. The prior spec's revision rules (update by matching ID, max-ID constraint) are retired.

---

## NPC reasoning entry persistence
**V1 said:** NPC Generator with extra retrieval turns: keep intermediate progress in the transient Console; do not save it as part of the NPC record set.

**V2 said:** Use a console-like feed to render agent chatter for intermediate steps.

**Actual:** Reasoning entries (kind=`reasoning`) are persisted as `SessionEntry` rows in the database and displayed in the NPC exchange feed. They are not transient. They are not part of the NPC card records (the `npcs` table), but they are durable session entries. The V2 "console-like feed" language was interpreted as a durable per-session feed, not the transient Console panel.

**Merged spec:** NPC reasoning/tool-call entries are persisted as session entries and are visible in the NPC exchange feed. They are not part of the NPC card set. The V1 instruction to route them only to the transient Console is retired.

---

## NPC exchange feed layout
**V1 said:** NPC output appears in the NPCs tab; intermediate progress (reasoning) goes to the Console.

**V2 said:** NPC mode uses a console-like feed to render agent chatter for intermediate steps. NPC cards are linked to a session but displayed across all sessions.

**Actual:** The NPC Cards tab is a vertically split panel. The top panel renders the NPC cards grid (paginated, filterable, session-aware). The bottom drawer renders the exchange feed for the active session, showing user prompts, reasoning steps (extracted `<thinking>` blocks), and notes (extracted `<notes>` blocks). The raw `<npcs>` XML is parsed out and not rendered in the feed — the cards grid serves that purpose.

**Merged spec:** The NPC Cards tab is a split panel: cards grid on top, exchange feed drawer on bottom. The exchange feed shows the active session's runs in order. Response entries are parsed XML: the `<thinking>` block is rendered with reasoning styling; the `<notes>` block is rendered with response styling; the `<npcs>` block is not rendered in the feed.

---

## NPC response envelope
**V1 said:** "Final output must conform to the structured NPC-card shape before being saved." Did not specify a wire format.

**V2 said:** (inherits V1 for this)

**Actual:** The NPC response is a structured XML envelope. Required tags: `<response>`, `<response-title>`, `<npcs>` (containing one or more `<npc>` blocks), `<notes>`. Each `<npc>` block contains `<name>`, `<bio>`, `<description>` (required) and optional `<species>`, `<ethnicity>`, `<gender>`, `<role>`, `<age>`. On the first run in a session, a `<session-title>` tag is also required. A single repair attempt is issued for malformed responses; if repair fails, the run fails without saving NPC data.

**Merged spec:** The NPC response uses the XML envelope described above. Repair behavior is one attempt; failure clears the run without touching stored NPC records.

---

# Interaction

## Enter key to submit
**V1 said:** Enter submits the active prompt. Shift+Enter inserts a new line.

**V2 said:** (inherits V1)

**Actual:** Ctrl+Enter submits the active prompt. The prompt textarea handles Enter as a normal newline (HTML default). Ctrl+Enter is disabled while a run is in progress or the session is a legacy NPC session, matching the Submit button's disabled state.

**Merged spec:** Enter inserts a newline in the prompt textarea. Submission is via the Submit button or the Ctrl+Enter keyboard shortcut. Both are disabled while a run is in progress or the session is locked. The V1 plain-Enter-to-submit behavior is retired.

---

# Session Management

## Explicit session creation
**V2 said:** There is a POST endpoint to create a new session (listed as one of the API calls).

**Actual:** There is no separate session creation endpoint. Sessions are created atomically inside the first POST /api/v2/runs call when no `sessionId` is provided. The client maintains a UI-local temporary session (sentinel ID `__temp__`) until the first run succeeds, then promotes it to the real persisted session ID received in the run response.

**Merged spec:** Sessions are created implicitly on the first run POST. There is no standalone session creation endpoint. The client may hold a temporary local session before the first run; that temp state is replaced by the real session ID after the first successful run.

---

## Session list filter parameter
**V2 said:** GET for session summaries uses a path param to filter by mode.

**Actual:** GET /api/v2/sessions uses an optional query parameter `?mode=` to filter by session mode. No path parameter is used.

**Merged spec:** Session list filtering is via `?mode=` query parameter, not a path parameter.

---

## GET full info for current session
**V2 said:** One of the API calls is "GET full info for current session."

**Actual:** No such single endpoint exists. Full session data is assembled from two requests: GET /api/v2/sessions (list, with optional mode filter) and GET /api/v2/sessions/:sessionId/feed (ordered run and entry history). The client composes these.

**Merged spec:** There is no single "full session info" endpoint. Full session state is assembled from the session list and the session feed endpoints.

---

# V1 Invariants Still Holding

The following V1 invariants are confirmed unchanged in 1.0 and should be carried forward verbatim into the merged spec:

- Do not use prior transcripts as silent prompt memory.
- Do not clear app-owned corpus artifacts during routine startup or restart.
- Do not treat Additional Context as retrieval corpus evidence; do not cite it as a source.
- Do not present unsupported inference as directly sourced claims.
- Do not overwrite saved NPC cards with invalid structured output.
- Do not record authorization headers or API keys in diagnostic output.
- Do not advance persisted state until ingestion and retrieval output are in a trustworthy condition.
